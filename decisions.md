## D-008 | 2026-07-18 | ESPN API host: use lm-api-reads, not fantasy.espn.com
Status: ACCEPTED

Alternatives rejected: driving a real browser (Playwright/Selenium) to solve the WAF challenge — large complexity/fragility increase for a problem that a different, equally-official ESPN host already avoids; the third-party `espn-api` PyPI package — hand-rolled requests using verified field paths keep the codebase consistent with the Sleeper/FFC readers and avoid an unverified dependency's version-compatibility risk.
Judged by: Phase 1 exit checklist (agent tool layer / ESPN reader works); every future ESPN pull silently succeeding rather than 202-ing forever.

## D-013 | 2026-07-19 | Bootstrap CIs built and verified (Phase 2 exit checklist complete) -- and they QUALIFY D-011's "ECR beats ADP at QB/TE" finding
Status: ACCEPTED
Context: phase2-backtest-harness-design.md §5 requires bootstrap confidence intervals on Spearman deltas ("report the CI, not just the point delta") and the Phase 2 exit checklist's last remaining item was "Bootstrap CI implementation verified against a hand-checked example." Built `backtest/bootstrap_ci.py`: `bootstrap_spearman_delta()` resamples the shared player universe (same resampled index applied to both systems and the actual outcome, so the delta is computed on matched pairs, not independently-resampled series) for ~2,000 iterations and reports percentile CIs. Verified against a hand-checked example BEFORE trusting it on real data: a 10-player synthetic set where candidate_ranks == actual_ranks exactly (a perfect predictor) and baseline_ranks is the exact reverse -- since resampling preserves pairwise equality, Spearman must be EXACTLY 1.0 and -1.0 respectively in every one of 2,000 iterations, so the CI must collapse to exact single points [1.0,1.0] / [-1.0,-1.0] / delta [2.0,2.0], not approximately. Asserted precisely in code, not eyeballed -- passed.
Then applied it for real: CI on the ECR-vs-ADP-order Spearman delta, all 4 positions, both worlds (`backtest/scorecards/bootstrap_ci_ecr_vs_adp.json`). Note a methodology difference from D-011: the bootstrap uses the INTERSECTION of both systems' own top-N universes (a fair paired comparison -- same exact players scored by both systems on every resample), whereas D-011's point estimates scored each system against its OWN top-N-by-its-own-rank universe independently (which can include slightly different players when ECR and ADP disagree on who belongs in the top 24/48/60/24). This is why the raw numbers differ slightly (e.g. 2024 TE: D-011's independent-universe delta was +0.142 favoring ECR; the shared-universe bootstrap point estimate is +0.032) -- the intersection approach is the more rigorous one for isolating "do these two systems actually rank the same players differently," and should be treated as the more trustworthy of the two going forward.
Decision: Record the real finding, even though it undercuts D-011's framing: **every single position/world's 95% CI on the ECR-vs-ADP-order delta straddles zero** (2024: QB [-.09,.10], RB [-.01,.14], WR [-.10,.06], TE [-.06,.13]; 2025: QB [-.10,.19], RB [-.07,.11], WR [-.08,.09], TE [-.14,.22]). At these universe sizes (21-56 players), NEITHER system is statistically distinguishable from the other at any position in either world -- D-011's "ECR beats ADP-order at QB and TE" should be read as a real but statistically UNCONFIRMED point-estimate pattern, not an established fact. This is exactly the "resulting" trap phase2 §5's variance protocol exists to catch: a real point-estimate gap that looks meaningful until you ask how much of it could be sampling noise at this sample size.
Alternatives rejected: skipping the real-data application and only shipping the hand-checked verification (rejected -- the whole point of building this now, ahead of Phase 3, is to have it ready and already caught a real premature conclusion in this project's own decision log, which is a stronger justification for building it now than an abstract "will need this eventually").
Judged by: phase2 §9 exit checklist (now 6/6 complete). Any future SM2 comparison (agent vs. baseline, agent vs. agent) must report a CI alongside the point delta from here forward, not repeat D-011's point-estimate-only framing.
Charter impact: none directly, but this is the operational precedent for how every future "beats the market" claim (O3, SM2) must be reported -- point estimate AND CI, never one without the other.
